# Agentic Software Engineering: Foundational Pillars and a Research Roadmap

**Publication Date:** 2025-09-23
**Link**: https://arxiv.org/abs/2509.06216v2
**Authors/Institution:** Queen’s University Canada, Huawei Canada, Concordia University Canada, Nara Institute of Science and Technology Japan

## Actionable Insights

### 1. The "Speed vs. Trust" Gap in Agentic Code Generation
* **Insight**: While autonomous coding agents enable massive productivity gains, they often produce code that is not truly "merge-ready." Agent-generated code frequently contains superficial fixes, subtle regressions, or lacks engineering hygiene, causing a bottleneck where human developers are overwhelmed by the verification process. 
* **Action**: Do not rely purely on automated test suites to validate agentic output. Organizations must shift from informal prompting to structured engineering practices that enforce architectural context, systemic quality, and auditable proof of correctness before code is merged.

### 2. The Code Review Bottleneck
* **Insight**: The hyper-productivity of agents generates a massive volume of code. Currently, over 68% of agent-generated pull requests (PRs) experience severe delays or remain completely unreviewed because human reviewers cannot keep pace with the machine-driven output.
* **Action**: Organizations must implement scalable, automated review pipelines or require agents to submit structured "Merge-Readiness Packs" (MRPs) containing explicit, evidence-backed deliverables to streamline human-in-the-loop validation.

### 3. Harnessing Parallel Generation (N-Version Programming)
* **Insight**: The Agentic era makes the historically expensive practice of "N-version programming" highly viable. A single natural-language specification can now trigger a team of autonomous agents to asynchronously generate multiple distinct solutions in parallel.
* **Action**: Optimize workflow orchestration to utilize inference-time compute. Have agents draft multiple competing pull requests per task, allowing the human "Agent Coach" to evaluate, select, or refine the most promising architecture before advancing to the review stage.

### 4. Transitioning to Declarative Specifications ("FMware")
* **Insight**: The paradigm of building software is shifting from manually coding low-level logic to declaratively describing behavior in natural language, effectively creating "FMware" that can be iteratively adapted without deep code rewrites.
* **Action**: Train senior engineering teams in "Agent Coaching." Emphasize skills in drafting precise, high-level intent, strategic workflows, and constraints, shifting their primary output from writing code blocks to authoring rigorous, machine-readable specifications.

## Core Concepts & Patterns & Taxonomy

### 1. The Four Pillars of Structured Agentic Software Engineering (SASE)
The SASE framework demands a radical reimagining of the four foundational pillars of Software Engineering, moving away from ad-hoc AI usage toward a structured duality (SE for Humans vs. SE for Agents).

| Pillar | Traditional Paradigm (Ad-hoc AI Coding) | Agentic SE Paradigm (SASE) | The Core Change & Impact |
| :--- | :--- | :--- | :--- |
| **1. Actors (The Team)** | Human-centric teams where developers write the code with AI as an autocomplete assistant. | **Hybrid Units:** Humans act as "Agent Coaches" while AI agents operate as autonomous executors. | Shift from writing code to orchestration and review. Humans handle strategy/mentoring; AI handles goal-oriented task completion. |
| **2. Processes (The Workflow)** | Unstructured, conversational chat prompts (e.g., trial-and-error chatting). | **Structured Activities:** Formal protocols for collaboration, including "agent-initiated handovers" when AI gets stuck. | Shift from unpredictable chat to rigorous, repeatable workflows that safely manage complex trade-offs and ambiguity. |
| **3. Artifacts (The Output)** | Transient, disposable text prompts that are lost after the code is generated. | **Durable Contracts:** Machine-readable artifacts like BriefingScripts, Consultation Request Packs, and Merge-Readiness Packs. | Prompts become institutional memory and verifiable proof of work, ensuring the agent's logic can be audited by humans. |
| **4. Tools (The Workspace)** | The standard human-centric Integrated Development Environment (IDE). | **Dual Workbenches:** Agent Command Environment (ACE) for human oversight, and Agent Execution Environment (AEE) for AI. | The workspace is split to serve two distinct users: a dashboard for human managers and a secure sandbox for AI workers. |

### 2. Deep Dive: Dual Tooling Workbenches (Pillar 4)
The traditional IDE is replaced by two purpose-built workbenches tailored to the cognitive strengths of their respective users:
* **Agent Command Environment (ACE)**: A command center optimized for human cognition. It allows the Agent Coach to specify intent, orchestrate tasks, observe agent activities, track costs, and review evidence-backed deliverables.
* **Agent Execution Environment (AEE)**: A digital workspace native to agents, optimized for high-speed compute, parallel execution, secure sandboxing, and continuous autonomous operation.

### 3. Deep Dive: Taxonomy of Structured Collaboration Artifacts (Pillar 3)
To replace transient, informal prompts, SASE introduces durable, version-controlled artifacts to manage the human-agent contract:
* **BriefingScript**: A human-authored mission plan outlining goals.
* **LoopScript**: A workflow playbook dictating how tasks should be executed.
* **MentorScript**: A human-authored guide ensuring agents follow institutional best practices and tribal knowledge.
* **Consultation Request Pack (CRP)**: An agent-generated artifact used to proactively invoke human expertise when the agent faces ambiguity or architectural trade-offs.
* **Merge-Readiness Pack (MRP)**: An agent-generated, evidence-backed deliverable presented to humans for final review.
* **Version Controlled Resolutions (VCRs)**: Formal, auditable human responses to CRPs or MRPs, ensuring the collaboration loop is traceable.

### 4. Hierarchical Framework for AI in SE (Autonomy Levels)
Borrowing from autonomous driving (SAE levels), the research outlines a taxonomy of AI integration in Software Engineering:
* **Level 0 (Manual Coding / SE 1.0)**: No AI. Humans write all code manually.
* **Level 1 (Token Assistance / SE 1.5)**: AI predicts immediate tokens (e.g., standard IDE auto-complete).
* **Level 2 (Task-Agentic / SE 2.0)**: AI maps a planned code description to a generated block of code (e.g., GitHub Copilot).
* **Level 3 (Goal-Agentic / SE 3.0)**: AI takes a broad technical goal and independently executes a multi-step plan to implement changes across a codebase (e.g., Devin, Claude Code).
* **Level 4 (Specialized Domain Autonomy / SE 4.0)**: AI maps broad technical mandates within a deep, specialized domain—specializing either in a specific tech stack (e.g., Frontend) or a specific quality attribute (e.g., Security).
* **Level 5 (General Domain Autonomy / SE 5.0)**: A conceptual stage where AI can generalize and adapt its high-autonomy capabilities to any unfamiliar technical challenge, tech stack, or domain.

### 5. Agency vs. Autonomy (Agent Types)
The research differentiates agent systems across a spectrum based on their capacity for self-governance:
* **Workflow Agents (High Agency, Low Autonomy)**: Systems where the orchestration, high-level logic, and tool invocations are explicitly hardcoded by humans. They execute predefined plans well but cannot independently adapt.
* **Autonomous Agents (High Autonomy)**: Systems that are given a high-level goal and independently formulate their own path, reasoning, planning, and invoking tools dynamically to achieve it.
